# User Manager — Автентифікація та авторизація

🇬🇧 [English version](../user-manager-auth.md)

**Модуль:** `system_modules/user_manager/`  
**Маршрути:** `/api/ui/modules/user-manager/`  
**Тип:** SYSTEM (in-process, без порту)

---

## Огляд

User Manager відповідає за управління ідентифікацією та контроль доступу в SelenaCore. Надає:

- **Токен пристрою** (device token) — довготривала автентифікація (HttpOnly cookie + заголовок)
- **Підвищені сесії** (elevated session) для чутливих операцій (підтвердження через PIN або QR, діє 10 хв)
- **Повний CRUD користувачів** з ролями: `owner` → `admin` → `user` → `guest`
- **QR-реєстрація пристрою** (новий браузер) та **QR-розблокування кіоска** (elevate)
- **Підтвердження PIN** для швидкого підвищення без QR

Усі токени генеруються випадково, зберігаються у вигляді SHA-256 хешу у SQLite (`/var/lib/selena/selena.db`).  
Шлях до бази даних задається як **абсолютний** через `sqlite+aiosqlite:////var/lib/selena/selena.db`  
(4 слеші = абсолютний шлях; 3 слеші = відносний від CWD контейнера).

## Модель автентифікації

```
Браузер / Телефон
      │
      │  X-Device-Token: <device_token>    (заголовок)
      │  Cookie: selena_device=<device_token>  (HttpOnly)
      │
      ▼
 user_manager → DeviceManager.verify(token)
      │
      ├─ токен дійсний  → повертає {user_id, role, display_name, ...}
      └─ токен недійсний → 401 Unauthorized


Чутлива операція (PATCH /users, DELETE /users, зміна PIN тощо)
      │
      │  X-Elevated-Token: <elevated_token>   (заголовок)
      │
      ▼
 _require_elevated() перевіряє:
      ├─ токен дійсний + не протермінований (< 10 хв) → виконати
      └─ відсутній / протермінований → 403 Forbidden
```

### Типи токенів

| Токен | Зберігається | Термін дії | Призначення |
|-------|-------------|-----------|-------------|
| `device_token` | HttpOnly cookie + `localStorage('selena_device')` | 30 днів | Ідентифікує зареєстрований браузер/телефон |
| `elevated_token` | Лише в `sessionStorage('selena_elevated')` | 10 хв | Дозволяє чутливі операції |
| `qr_session` | In-memory словник у user-manager | 5 хв (`_QR_TTL = 300`) | Одноразове QR-рукостискання |

## Ролі

| Роль | Рівень | Може |
|------|--------|------|
| `owner` | 4 | Все, включно з видаленням адміністраторів |
| `admin` | 3 | Управляти користувачами (крім owner), переглядати аудит-лог |
| `user` | 2 | Власний профіль, зміна свого PIN |
| `guest` | 1 | Лише читання, без чутливих операцій |

Налаштування прав для кожної ролі зберігається в таблиці `permissions` та може бути змінено через `PATCH /roles/{role}` (тільки для owner з підвищеними правами).

## Реєстрація пристрою (новий браузер/телефон)

Новий пристрій реєструється, вказуючи ім'я користувача та PIN.

### Варіант A — Пряма реєстрація (AuthWall)

```
Браузер                       user-manager
   │                               │
   │── POST /auth/register ────────►│
   │   {username, pin, device_name} │
   │                               │── перевірка облікових даних
   │                               │── створення запису пристрою
   │◄── 201 {device_token} ─────────│
   │    Set-Cookie: selena_device   │
```

### Варіант Б — QR-реєстрація (сканування телефоном)

```
Браузер ПК                    user-manager               Телефон
      │                            │                        │
      │── POST /auth/qr/start ────►│                        │
      │   {mode:"access"}          │── створення сесії ─────│
      │◄── {session_id,            │   термін 5 хв          │
      │     qr_image,              │                        │
      │     join_url,              │   QR містить LAN IP*   │
      │     expires_in:300}        │                        │
      │                            │                        │
      │  [показує QR на екрані]    │    [Телефон сканує]    │
      │                            │                        │
      │                            │◄── GET /auth/qr/join/{id}  ─── Браузер телефону
      │                            │    → qr_join.html
      │                            │◄── POST /auth/qr/complete/{id}
      │                            │    {username, pin, device_name}
      │                            │── перевірка + device_token для телефону
      │                            │── сесія → "complete"
      │                            │
      │── GET /auth/qr/status/{id}►│  (опитування кожні 2с)
      │◄── {status:"complete",     │
      │     device_token:"..."}    │
```

`*` **Визначення LAN IP:** коли `qr_start` викликається з localhost/127.0.0.1 (кіоск), сервер визначає реальний LAN IP через UDP-зонд і підставляє його в `join_url`. Це гарантує, що QR містить досяжну адресу (`http://192.168.1.x/...` замість `http://localhost/...`).

## Розблокування кіоска через QR (elevate)

На екрані пристрою `KioskElevationGate` перекриває обмежені розділи. Два способи розблокування:

### Розблокування через PIN

```
Кіоск                         user-manager
   │                               │
   │── POST /auth/pin/confirm ─────►│
   │   X-Device-Token: <token>      │── перевірка токену пристрою
   │   {pin}                        │── перевірка PIN-хешу
   │◄── 200 {elevated_token} ───────│
   │    (сесія 10 хв)              │
```

### Розблокування через QR

```
Кіоск                         user-manager               Телефон
   │                               │                        │
   │── POST /auth/qr/start ────────►│                        │
   │   {mode:"elevate"}            │── створення сесії ─────│
   │◄── {session_id,               │   mode="elevate"       │
   │     qr_image (LAN IP),        │   термін 5 хв          │
   │     expires_in:300}           │                        │
   │                               │   QR на екрані кіоска  │
   │  [кіоск показує QR+таймер]   │    [Телефон сканує]    │
   │                               │                        │
   │                               │◄── GET /auth/qr/join/{id}  ─── Телефон
   │                               │    → qr_join.html (fetches /qr/info)
   │                               │      показує "Підтвердити доступ?"
   │                               │◄── POST /auth/qr/approve/{id}
   │                               │    X-Device-Token: <phone_token>
   │                               │── перевірка токену телефону
   │                               │── видача elevated_token
   │                               │── сесія → "complete"
   │                               │
   │── GET /auth/qr/status/{id} ──►│  (опитування кожні 2с)
   │◄── {status:"complete",        │
   │     elevated_token:"..."}     │
   │                               │
   │  [замок знімається]           │  [Телефон: "Авторизацію підтверджено"]
```

## Сторінка підтвердження QR (`/auth/qr/join/{session_id}`)

Самостійна HTML-сторінка, що роздається з `system_modules/user_manager/qr_join.html`.  
Розроблена для мобільних браузерів. Особливості:

| Функція | Деталі |
|---------|--------|
| Перевірка сесії | Спочатку запитує `GET /auth/qr/info/{id}` — визначає режим та час, що залишився |
| Таймер зворотного відліку | ХВ:СС, стає червоним при < 60 с, показує "Код прострочено" при 0 |
| Вже прострочений | 410 → одразу показує "QR-код прострочений", форма не відображається |
| Режим `elevate` | Зчитує токен телефону → "Підтвердити доступ?" → галочка при успіху |
| Режим `access` | Форма (ім'я користувача + PIN) → реєстрація пристрою → збереження токена в `localStorage` |
| Локалізація | EN / UK через `localStorage('selena-lang')`, вбудовані словники |
| BASE URL | Обчислюється з `window.location.pathname` — без хардкоду хостів або портів |

---

## Таймер зворотного відліку

QR-сесії дійсні **5 хвилин** (`_QR_TTL = 300`).

- **Сторінка телефону:** таймер у заголовку сторінки (ХВ:СС), червоніє при 60 с
- **QrPane кіоска:** таймер під QR-зображенням (моноширинний, tabular-nums), червоніє при 60 с
- **При закінченні:** сервер повертає HTTP 410; обидві сторони показують "QR прострочено" + кнопку "Згенерувати новий"
- **TTL налаштовується** через `_QR_TTL` у `system_modules/user_manager/module.py`

## Довідник API

Базовий шлях: `/api/ui/modules/user-manager`

### Автентифікація — реєстрація пристрою

| Метод | Шлях | Auth | Опис |
|-------|------|------|------|
| `POST` | `/auth/register` | — | Реєстрація нового пристрою (username + PIN) |
| `POST` | `/auth/pin/confirm` | токен пристрою | Перевірка PIN → отримання elevated_token |
| `POST` | `/auth/device/verify` | токен пристрою в заголовку | Перевірка токена, повертає інфо про користувача |
| `POST` | `/auth/device/revoke` | токен пристрою | Відкликання власного токена |
| `GET`  | `/auth/devices` | токен пристрою | Список зареєстрованих пристроїв власного облікового запису |

### Автентифікація — QR-флоу

| Метод | Шлях | Auth | Опис |
|-------|------|------|------|
| `POST` | `/auth/qr/start` | — | Створення QR-сесії (`mode`: `access` або `elevate`) |
| `GET`  | `/auth/qr/status/{id}` | — | Опитування статусу сесії |
| `GET`  | `/auth/qr/info/{id}` | — | Режим + `expires_in_seconds` (для сторінки телефону) |
| `GET`  | `/auth/qr/join/{id}` | — | Сторінка підтвердження на телефоні (HTML) |
| `POST` | `/auth/qr/approve/{id}` | токен пристрою | Телефон підтверджує розблокування кіоска (режим elevate) |
| `POST` | `/auth/qr/complete/{id}` | — | Телефон відправляє username+PIN (режим access) |

#### `POST /auth/qr/start` — Запит

```json
{ "mode": "access" }   // реєстрація нового браузера
{ "mode": "elevate" }  // розблокування кіоска
```

#### `POST /auth/qr/start` — Відповідь `201`

```json
{
  "session_id": "uuid-...",
  "join_url":   "http://192.168.1.45/api/ui/modules/user-manager/auth/qr/join/uuid-...",
  "qr_image":   "data:image/png;base64,...",
  "expires_in": 300
}
```

`join_url` завжди містить **LAN IP** якщо запит надійшов із localhost/127.0.0.1.

#### `GET /auth/qr/info/{id}` — Відповідь `200`

```json
{
  "mode": "elevate",
  "status": "pending",
  "expires_in_seconds": 247
}
```

Повертає `410 Gone` якщо сесія прострочена.

#### `GET /auth/qr/status/{id}` — Відповідь `200`

```json
// очікуємо:
{ "status": "pending" }

// завершено (режим access):
{ "status": "complete", "device_token": "...", "user_id": "uuid-..." }

// завершено (режим elevate):
{ "status": "complete", "elevated_token": "...", "user_id": "uuid-..." }
```

Повертає `404` якщо сесія не знайдена, `410` якщо прострочена.

## KioskElevationGate

React-компонент (`src/components/KioskElevationGate.tsx`), що перекриває обмежені розділи UI на екрані кіоска.

**Умови визначення кіоска:**
- Hostname: `localhost` або `127.0.0.1`
- URL містить `?kiosk=1`

**Обмежені маршрути**: `/modules`, `/system`, `/integrity`, `/settings`  
**Завжди доступно**: `/` (дашборд)

**Поведінка:**
1. Користувач переходить до обмеженого розділу
2. Оверлей замка перекриває повний екран
3. Розблокування через вкладку PIN або QR
4. Elevated token зберігається в `sessionStorage` + Zustand store
5. При успіху: оверлей зникає, розділ відкривається
6. При закінченні токена (10 хв, від сервера): замок з'являється знову

**Додатково на вкладці QR:** таймер зворотного відліку (ХВ:СС) під QR-кодом; червоніє при 60 с; при закінченні — кнопка "Згенерувати новий код".

## Нотатки з безпеки

- `device_token` генерується через `secrets.token_urlsafe(48)` — 64 символи base64url
- У базі даних зберігається лише SHA-256 хеш токена
- `elevated_token` — окремий короткочасний токен, не перевикористовується
- QR-сесії зберігаються лише в пам'яті; перезапуск очищує всі очікуючі сесії
- `qr_join.html` зчитує токен телефону з `localStorage` / cookie — токен **ніколи** не передається через URL
- PATCH/DELETE по користувачах вимагає одночасно дійсного device_token **та** elevated_token
- Захист від брутфорсу PIN: rate-limiting middleware (5 спроб → блок 10 хв) на рівні middleware API
