# Менеджер резервного копіювання

Системний модуль `backup_manager` створює локальні tar.gz-архіви стану
SelenaCore. Перша фаза охоплює ручні та заплановані локальні копії,
відновлення з запобіжниками та інтерфейс налаштувань. Хмарне E2E-відвантаження
й перенос секретів через QR-код присутні в коді (`cloud_backup.py`,
`qr_transfer.py`), але поки не підключені.

## Що зберігається

Дві категорії, обидві потрапляють до `/var/lib/selena/backups/`:

| Категорія | Шляхи | Стандартно | Перемикач |
|---|---|---|---|
| `core` | `/var/lib/selena/selena.db`, `/var/lib/selena/widget_layout.json`, `/var/lib/selena/modules/`, `/etc/selena/` | увімкнено | завжди |
| `secrets` | `/secure/` (без `vault_key`) | увімкнено | опційно |

База SQLite копіюється через [SQLite Online Backup API](https://www.sqlite.org/backup.html)
до тимчасового файлу перед додаванням в архів — тож копія, зроблена під
час запису selena-core, не містить розірваної бази.

`/secure/vault_key` **завжди виключається**: відновлення vault-ключа з
одного пристрою на інший зруйнувало б криптографічну ізоляцію per-device.
Vault регенерується при першому запуску після відновлення.

Голосові моделі (`/var/lib/selena/models/`) і записи свідомо не входять
до жодної категорії — вони великі та завантажуються наново.

## Ручне резервування

Відкрийте сторінку *Налаштування* модуля на дашборді:

1. Відмітьте категорії, які хочете включити (категорія Core завжди увімкнена).
2. Натисніть **Створити зараз**. Архів з'явиться у списку *Резервні копії*
   за кілька секунд.

Через REST:

```bash
curl -X POST https://<host>/api/ui/modules/backup-manager/backup/create
```

Відповідь містить ім'я архіву, розмір та SHA-256.

## Запланований бекап

У картці *Розклад* оберіть один з варіантів:

- **Вимкнено** — без автоматичних копій.
- **Щодня** о HH:MM (за замовчуванням 03:00).
- **Щотижня** у неділю о HH:MM.
- **Власний cron** — будь-який вираз, який підтримує модуль `scheduler`,
  наприклад `cron:30 4 * * *` чи `every:6h`.

Збереження публікує подію `scheduler.register` з `job_id` =
`backup-manager.scheduled` та `event_type` = `backup.scheduled.fire`.
Модуль резервування підписаний на цю подію і викликає `create_backup()`
при її надходженні. Вимкнення публікує `scheduler.unregister`.

## Відновлення

1. У списку *Резервні копії* натисніть **Відновити** на потрібному архіві.
2. Модальне вікно попередить про перезапис даних. Введіть ім'я архіву
   для підтвердження.
3. Перед відновленням створюється знімок поточного стану
   `selena_prerestore_<timestamp>.tar.gz` (окремий пул ротації, типово 3 копії).
4. Обраний архів розпаковується в `/`. Будь-які елементи з виходом за
   межі цільової директорії відкидаються до початку розпаковки.
5. Після успішної розпаковки модуль виконує `systemctl restart selena-core`,
   щоб нова база застосувалася без ручного втручання. Якщо `systemctl`
   недоступний (середовище розробки), відповідь повідомить про це й
   оператор має перезапустити сервіс інакше.

Якщо потрібно відкотити саме відновлення — pre-restore-знімок лежить у
тій самій директорії, відновлюється через той самий потік.

## Структура файлів

```
/var/lib/selena/
├── selena.db                           ← через Online Backup API
├── widget_layout.json
├── modules/
│   └── backup_manager/
│       └── settings.json               ← вибір користувача (категорії, розклад)
└── backups/
    ├── selena_backup_YYYYMMDDTHHMMSSZ.tar.gz       ← звичайний пул
    └── selena_prerestore_YYYYMMDDTHHMMSSZ.tar.gz   ← пул pre-restore
/etc/selena/                            ← конфігурації
/secure/                                ← OAuth-токени, креди модулів (без vault_key)
```

Ротація — окрема для кожного пулу: `max_backups` (типово 5, діапазон
1–50) для звичайних копій; `PRERESTORE_RETENTION` (типово 3) для
pre-restore-знімків.

## API

Усі ендпоінти змонтовані за префіксом `/api/ui/modules/backup-manager/`.

| Метод | Шлях | Опис |
|---|---|---|
| GET | `/config` | Поточні налаштування |
| PATCH | `/config` | Зберегти налаштування; перереєструє розклад |
| GET | `/list` | Усі архіви (звичайні + pre-restore) з розміром і часом |
| POST | `/backup/create` | Ручний бекап з поточними категоріями |
| POST | `/backup/{name}/restore` | Тіло `{"confirm": "<name>"}` обов'язкове |
| DELETE | `/backup/{name}` | Видалити архів |
| GET | `/backup/{name}/download` | Стрім `.tar.gz` |
| POST | `/backup/upload` | Multipart-завантаження зовнішнього `.tar.gz` |
| GET | `/widget/data/state` | Зарезервовано для майбутнього віджета |
| GET | `/settings` | HTML-сторінка налаштувань |

`/status`, `/backup/local` і `/restore/local` залишені як legacy-аліаси.

## Конфігурація

Зміна через змінні оточення:

| Змінна | Типово | Призначення |
|---|---|---|
| `BACKUP_DEST` | `/var/lib/selena/backups` | Куди писати архіви |
| `BACKUP_MANAGER_STATE_DIR` | `/var/lib/selena/modules/backup_manager` | Розташування `settings.json` |
| `PRERESTORE_RETENTION` | `3` | Скільки `selena_prerestore_*` зберігати |

## Поза цим етапом (фаза 2+)

- Хмарний E2E-бекап (`cloud_backup.py`) — потрібні налаштування пароля,
  URL платформи та UI.
- QR-перенос для bootstrap нового пристрою (`qr_transfer.py`).
- Голосові моделі та записи як окремі категорії.
- Віджет на дашборді — ендпоінт `/widget/data/state` уже є; залишається
  лише запис у маніфесті та реєстрація шаблону.
