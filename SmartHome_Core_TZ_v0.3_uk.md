## 1. Вступ та концепція

SelenaCore — це хаб розумного дому з відкритим кодом (MIT). Встановлюється на Raspberry Pi 4/5 або будь-який одноплатний комп'ютер під керуванням Linux. Для базової роботи підписка не потрібна. Інтегрується з платформою SmartHome LK для хмарних функцій, маркетплейсу модулів та віддаленого управління.

### 1.1 Три фундаментальні принципи

**Ядро незмінне** — файли ядра захищені контрольним хешем SHA256 та прапорцем `chattr +i`. Модифікація неможлива без явного оновлення через офіційний канал платформи.

**Модулі ізольовані** — системні модулі працюють у процесі через importlib. Користувацькі модулі працюють як підпроцеси та спілкуються виключно через WebSocket Module Bus (`ws://core:7070/api/v1/bus`). Прямий доступ до даних ядра та розділу `/secure` заборонено.

**Агент спостерігає** — незалежний процес `IntegrityAgent` постійно перевіряє SHA256 хеші файлів ядра та реагує за ланцюжком: зупинка модулів → сповіщення платформи → відкат із резервної копії → SAFE MODE.

### 1.2 Відкритий код

Проєкт розповсюджується за ліцензією MIT. ДБЖ/резервне живлення, кастомні апаратні конфігурації — на розсуд спільноти. Ця специфікація визначає базову функціональність ядра.

## 2. Архітектура: гібридна модульна система

Замість окремого Docker-контейнера на кожен модуль або монолітного менеджера плагінів використовується гібридна схема з двома типами модулів.

### 2.1 Два типи модулів

| Тип | Завантаження | Накладні витрати | Комунікація | Розташування |
|---|---|---|---|---|
| **Системні модулі** (`type=SYSTEM`) | У процесі через `importlib` | ~0 МБ на модуль | Прямий доступ до EventBus та бази даних | `system_modules/` |
| **Користувацькі модулі** (`type=UI/INTEGRATION/DRIVER/AUTOMATION/IMPORT_SOURCE`) | Підпроцеси в Docker-контейнерах | Накладні витрати на контейнер | WebSocket Module Bus за адресою `ws://core:7070/api/v1/bus?token=TOKEN` | Встановлюються через маркетплейс |

**Системні модулі** завантажуються безпосередньо в процес ядра. Вони не мають накладних витрат Docker, ні витрат на підпроцеси, і можуть реєструвати FastAPI-роутер, змонтований за адресою `/api/ui/modules/{name}/`. Поставляються разом з ядром і не можуть бути видалені.

**Користувацькі модулі** працюють як підпроцеси всередині Docker-контейнерів. Вони не мають окремих портів — весь трафік проходить через єдину точку WebSocket Module Bus. Збій модуля (Exception) перехоплюється, логується, і перезапускається лише контейнер цього модуля. OOM/segfault призводить до перезапуску контейнера через Docker-політику `--restart=unless-stopped`.

### 2.2 Module Bus (за мотивами CAN-bus)

Module Bus — це єдиний канал комунікації між користувацькими модулями та ядром. Натхненний архітектурою CAN-bus: єдина спільна шина, де ядро виступає головним вузлом.

**Точка підключення:** `ws://core:7070/api/v1/bus?token=TOKEN`

**Типи повідомлень:**

| Тип повідомлення | Напрямок | Призначення |
|---|---|---|
| `announce` | Модуль → Ядро | Модуль реєструється при підключенні |
| `intent` | Ядро → Модуль | Ядро передає інтент користувача модулю |
| `intent_response` | Модуль → Ядро | Модуль повертає результат інтенту |
| `event` | Двонаправлений | Доставка подій EventBus через шину |
| `ping` / `pong` | Двонаправлений | Підтримка з'єднання та перевірка стану |
| `api_request` | Модуль → Ядро | Модуль викликає API-ендпоінт ядра |
| `api_response` | Ядро → Модуль | Ядро повертає відповідь API |
| `shutdown` | Ядро → Модуль | Сигнал коректного завершення |

**Подвійні канали на з'єднання:**

- **Критична черга** — ємність 100 елементів, зворотний тиск (повільний споживач блокує відправника). Використовується для інтентів, API-запитів, завершення роботи.
- **Черга подій** — ємність 1 000 елементів, при переповненні видаляються найстаріші. Використовується для подій, некритичних сповіщень.

**Надійність:**

- Circuit breaker на кожен модуль: 5 збоїв за 60 сек → модуль позначається як несправний → перепідключення з експоненціальною затримкою
- ACL-дозволи: кожен модуль оголошує необхідні дозволи в `manifest.json`; шина забезпечує їх виконання під час роботи

### 2.3 EventBus

EventBus — це система pub/sub на основі asyncio.Queue (максимум 10 000 елементів), яка маршрутизує події всередині ядра.

- **DirectSubscription** — системні модулі підписуються з асинхронними зворотними викликами в процесі. Нульові витрати на серіалізацію.
- **Доставка через Module Bus** — події, що відповідають підпискам користувацького модуля, серіалізуються в JSON та надсилаються через WebSocket-шину через чергу подій.

### 2.4 Watchdog — дворівневий захист

- **Рівень 1 — systemd**: `smarthome-core.service` з `Restart=always`, `RestartSec=5s`
- **Рівень 2 — Docker**: `--restart=unless-stopped` на контейнерах користувацьких модулів
- **Integrity Agent**: окремий `smarthome-agent.service`, незалежний від контейнера ядра

### 2.5 Економія пам'яті порівняно з окремими контейнерами

| Конфігурація | RAM (типове навантаження) |
|---|---|
| Один контейнер на модуль (8 модулів) | ~1 200 МБ |
| Гібридна схема (ті ж 8 модулів: 10 системних + 8 користувацьких) | ~500 МБ |
| Економія | ~700 МБ (−58%) |

## 3. Класифікація модулів

### 3.1 Типи модулів

| Тип | Видаляється? | Опис |
|---|---|---|
| `SYSTEM` | Ні | Поставляється з ядром. Розширені привілеї. Працює в процесі ядра. |
| `UI` | Так | Іконка в меню + віджет на панелі + сторінка налаштувань. Пісочниця iframe. |
| `INTEGRATION` | Так | Зовнішні сервіси через OAuth/API. Токени в ядрі, модуль не бачить їх напряму. |
| `DRIVER` | Так | Драйвер протоколу: Zigbee, Z-Wave, MQTT, HTTP-пристрої. |
| `AUTOMATION` | Так | Сценарії без UI. Обробники подій + планувальник. Найлегший (~40 МБ). |
| `IMPORT_SOURCE` | Так | Імпорт з Home Assistant, Tuya, Philips Hue та інших систем. |

### 3.2 UI-профілі

| Профіль | Компоненти | Приклад |
|---|---|---|
| `HEADLESS` | Без UI | Нічний режим, будильник |
| `SETTINGS_ONLY` | Сторінка налаштувань | Системний модуль voice-core |
| `ICON_SETTINGS` | Іконка + налаштування | Інтеграція Gmail |
| `FULL` | Іконка + віджет + налаштування | Модуль клімату, модуль освітлення |

### 3.3 Режими роботи (manifest.json)

- `always_on` — працює постійно. UI-модулі, драйвери, Telegram-сповіщення.
- `on_demand` — підпроцес запускається, виконує завдання, зупиняється. AUTOMATION.
- `scheduled` — cron-рядок у маніфесті. Приклад: `"*/5 * * * *"` для перевірки Gmail кожні 5 хвилин.

### 3.4 manifest.json — Структура

```yaml
name:          my-module          # unique name (snake_case)
version:       1.0.0              # semver
type:          INTEGRATION        # module type
ui_profile:    ICON_SETTINGS      # UI profile
api_version:   "1.0"
runtime_mode:  scheduled
schedule:      "*/5 * * * *"
permissions:
  - device.read
  - events.subscribe
intents:
  - pattern: "check mail"
    action: "check_inbox"
  - pattern: "send email to {recipient}"
    action: "send_email"
publishes:
  - "email.received"
  - "email.sent"

# If ui_profile != HEADLESS:
ui:
  icon:     icon.svg
  widget:
    file:   widget.html
    size:   "2x1"                 # 1x1 | 2x1 | 2x2 | 4x1
  settings: settings.html

# If type is INTEGRATION:
oauth:
  provider: google                # google | telegram | custom
  scopes:
    - gmail.readonly
```

### 3.5 Безпека UI-компонентів

- Усі віджети та сторінки налаштувань рендеряться в `<iframe sandbox>` — модуль не має доступу до DOM ядра
- Комунікація лише через `window.postMessage` з білим списком дозволених типів повідомлень
- CSP-заголовок: `default-src 'self'` — інлайн-скрипти заборонені

## 5. Перший запуск — майстер налаштування

Мета: користувач без технічних знань налаштовує систему за 5-10 хвилин, використовуючи лише телефон.

### 5.1 Крок 0 — Перед увімкненням: запис образу на SD

- Готовий образ `.img` (SmartHome LK OS Lite) завантажується з веб-сайту платформи
- Записується через Raspberry Pi Imager або balenaEtcher — додаткове налаштування не потрібне
- Образ: Raspberry Pi OS Lite + Docker + smarthome-core попередньо встановлені

### 5.2 Крок 1 — Перше увімкнення: точка доступу + QR

При першому запуску (або якщо Wi-Fi не налаштований) ядро створює точку доступу:

```
SSID:     SmartHome-Setup
Password: smarthome
```

**Якщо підключений HDMI-дисплей:**
→ QR-код відображається на TTY1
→ Скануйте → відкривається майстер у браузері телефону

**Якщо дисплея немає (headless):**
→ Підключіться до SmartHome-Setup з телефону
→ Відкрийте браузер → `192.168.4.1`
→ Той самий майстер

Резервний варіант mDNS: `http://smarthome-setup.local`

### 5.3 Майстер — 9 кроків у браузері телефону

| # | Крок | Деталі |
|---|---|---|
| 1 | **Мова інтерфейсу** | Вибір: ru / uk / en. Впливає на весь текст та голоси TTS. |
| 2 | **Мережа Wi-Fi** | Список виявлених мереж. Введіть пароль. Pi підключається та перевіряє інтернет. |
| 3 | **Назва пристрою** | Наприклад, "Розумний дім — Кухня". Відображається на платформі та в голосових відповідях. |
| 4 | **Часовий пояс** | Вибір зі списку або автовизначення за IP. |
| 5 | **Голосова модель STT** | Vosk small (швидка, Pi 4) / medium (збалансована) / large (якісна, Pi 5). Завантажується. |
| 6 | **Голос TTS (Piper)** | Список голосів для обраної мови. Кнопка "Прослухати". Завантаження ~50 МБ. |
| 7 | **Перший користувач** | Ім'я адміна, PIN 4-8 цифр. Опціонально: голосовий відбиток (5 фраз). |
| 8 | **Реєстрація на платформі** | QR або посилання. Опціонально — можна пропустити, працює повністю локально. |
| 9 | **Імпорт (опціонально)** | Home Assistant / Tuya / Philips Hue. OAuth через посилання. Можна пропустити. |

### 5.4 Екран "Що далі" після майстра

- Підключити пристрої → `/discovery` (мережевий сканер)
- Встановити модулі → `/modules/install` (маркетплейс)
- Налаштувати голосовий асистент → `/settings/voice`
- Додати додаток на головний екран → кнопка "Встановити PWA"
- Документація та відео → `docs.selenehome.tech`

## 6. ОС та режими інтерфейсу

### 6.1 Рекомендовані операційні системи

| ОС | RAM у простої | Рекомендація |
|---|---|---|
| **Raspberry Pi OS Lite** | ~150 МБ | ✅ Рекомендовано. Офіційна, найкраща підтримка апаратного забезпечення Pi. |
| **DietPi** | ~90 МБ | ✅ Рекомендовано. Мінімалістична, вбудований інсталятор Docker. |
| Armbian | ~170 МБ | Для сторонніх одноплатників (Orange Pi, NanoPi, Rock Pi). |
| Ubuntu Server 24.04 | ~240 МБ | Альтернатива, якщо потрібна екосистема Ubuntu. |
| Raspberry Pi OS Desktop | ~500 МБ | ⚠️ Тільки якщо потрібен робочий стіл. ~350 МБ втрачено. |

### 6.2 Автовизначення режиму UI при запуску

Веб-сервер `:80` працює у всіх режимах постійно. Локальний дисплей — це додатковий клієнт.

| Режим | Умова | Опис |
|---|---|---|
| `HEADLESS` | Немає HDMI | Тільки веб-сервер. Доступ: smarthome.local:80 + Tailscale. |
| `KIOSK` | X11/Wayland + HDMI | `chromium --kiosk http://localhost:80` поверх робочого столу. |
| `FRAMEBUFFER` | Lite OS + HDMI + Chromium | `chromium --ozone-platform=drm` без X11, напряму у фреймбуфер. |
| `TTY` | Lite OS + HDMI, без Chromium | Python Textual TUI (~15 МБ) на TTY1. Статус + навігація. |

Алгоритм автовизначення (`core/ui_detector.py`):

```python
def detect_display_mode() -> str:
    # 1. Is X11/Wayland available?
    if os.environ.get('DISPLAY') or os.environ.get('WAYLAND_DISPLAY'):
        return 'kiosk'
    # 2. Is HDMI connected?
    hdmi = Path('/sys/class/drm').glob('*/status')
    if any('connected' in p.read_text() for p in hdmi):
        if shutil.which('chromium-browser'):
            return 'framebuffer'
        return 'tty'
    # 3. No display
    return 'headless'
```

### 6.3 PWA (Progressive Web App)

- `manifest.json` + Service Worker: ui-core підтримує встановлення як PWA
- **Офлайн-сторінка**: при відсутності з'єднання з Pi відображається останній стан з кешу
- **Іконка** на головному екрані телефону: нативний вигляд без панелі браузера
- **Web Push VAPID**: сповіщення на телефон навіть при закритому браузері (через `notify-push`)

### 6.4 Конфігурація UI (core.yaml)

```yaml
ui:
  web_port: 80
  display_mode: auto        # auto | headless | kiosk | framebuffer | tty
  mdns_announce: true       # smarthome.local
  tty_device: /dev/tty1
  framebuffer: /dev/fb0
  https: true               # self-signed certificate
```

## 7. Голосовий асистент та LLM

### 7.1 Компоненти voice-core

| Компонент | Стек | Характеристики |
|---|---|---|
| Wake-word | openWakeWord | < 5% CPU, завжди у фоні, настроюване слово активації |
| STT | Vosk small/medium/large | Обирається в майстрі. Тільки локально, інтернет не потрібен. |
| TTS | Piper neural | Вибір голосу в майстрі з попереднім прослуховуванням. Офлайн. Затримка ~300мс. |
| Speaker ID | resemblyzer | Реєстрація: 5 фраз → 256-float d-вектор у `/secure/biometrics/` |
| Режим приватності | GPIO + голос | Фізична GPIO-кнопка **АБО** команда "Дім, тихо" → мікрофон вимкнено |

### 7.2 Конвеєр голосового запиту

```
openWakeWord → чує слово активації
      ↓
Запис аудіо (до паузи 1.5 сек)
      ↓
Vosk → текст запиту                     ~0.3–1.5 сек
      ↓
Speaker ID: хто говорить?               ~200 мс
      ↓
Intent Router — Рівень 1: Fast Matcher  ~0 мс (пошук у YAML)
      ↓ не знайдено
Intent Router — Рівень 2: Системні      ~мкс (regex у процесі)
      ↓ не знайдено
Intent Router — Рівень 3: Module Bus    ~мс (користувацькі модулі через WebSocket)
      ↓ не знайдено
Intent Router — Рівень 4: Ollama LLM    ~3–8 сек (тільки Pi 5)
      ↓
Знайдено модуль → виконання
Не знайдено    → TTS: "Такого модуля немає. Пошукати в маркетплейсі?"
      ↓
Piper TTS → відтворення відповіді       ~300 мс
      ↓
Запис у історію діалогу (SQLite)
```

### 7.3 Intent Router — чотири рівні

**Рівень 1 — Fast Matcher (~0мс, працює на Pi 4 і Pi 5)**
- Статичні правила на основі ключових слів/regex для частих команд
- Налаштовується у YAML: `"увімкни світло" → lights.on`
- Без LLM, без диспетчеризації модулів — миттєвий пошук

**Рівень 2 — Інтенти системних модулів (мікросекунди, у процесі)**
- Системні модулі реєструють шаблони інтентів через декоратор `@intent`
- Зіставлення regex з текстом запиту в процесі
- Прямий виклик callback, без витрат на серіалізацію

**Рівень 3 — Інтенти Module Bus (мілісекунди, користувацькі модулі)**
- Користувацькі модулі оголошують `intents` у `manifest.json`
- Ядро зіставляє запит із зареєстрованими шаблонами
- Відправляється до відповідного модуля через WebSocket-повідомлення `intent`
- Модуль повертає `intent_response` з результатом

**Рівень 4 — LLM-інтент (3-8 сек, тільки Pi 5 з 8ГБ RAM)**
- Ollama з моделлю phi-3-mini (3.8B int4) або gemma-2b
- Системний промпт містить динамічний реєстр встановлених модулів
- Реєстр перебудовується при кожному встановленні/видаленні модуля
- Повертає JSON: `{ intent, module, params, confidence }`
- Якщо `confidence < 0.7` → просить повторити
- Автоматично вимикається при вільній RAM < 5ГБ

### 7.4 Голосове введення через браузер клієнта

- `getUserMedia()` → WebSocket → Pi: аудіо передається фрагментами по 100мс
- Pi: `ffmpeg` → WAV 16kHz → Vosk → Intent Router → Piper TTS → WAV-відповідь
- Нічого не йде в хмару — весь конвеєр працює локально на Pi
- Автовизначення мікрофона клієнта: `enumerateDevices()` — якщо немає, кнопка PTT прихована

### 7.5 Мовні налаштування

- Мова інтерфейсу та мова голосу TTS обираються незалежно
- Підтримувані бета-мови: `ru`, `uk`, `en`
- Додавання мови = завантаження мовного пакету Piper (~50 МБ) через `/settings/voice`
- Системний промпт LLM надсилається мовою активного користувача

### 7.6 Біометрія — абсолютне обмеження

> **Голосові відбитки (d-вектор) та ембединги обличчя зберігаються ТІЛЬКИ у `/secure/biometrics/` на пристрої. Хмарна синхронізація заблокована на рівні ядра. Це обмеження не налаштовується і не може бути зняте жодною командою платформи.**

## 9. Користувачі, авторизація, аудит

### 9.1 Модель користувачів

Плоска модель користувачів без рольових дозволів:

- **admin** — перший користувач, створений під час майстра налаштування (має PIN)
- **resident** — всі наступні користувачі (мешканці дому, ім'я + опціональна прив'язка пристрою)

Шлюз підвищення привілеїв PIN/QR — єдиний бар'єр безпеки. Будь-який підвищений користувач
(будь-хто, хто ввів правильний PIN або сканував QR) може отримати доступ до всіх налаштувань,
керувати користувачами, модулями та пристроями. Немає матриці дозволів за ролями.

### 9.2 Методи авторизації в ui-core

- **PIN** (4-8 цифр) — завжди доступний
- **Face ID** — якщо зареєстрований і клієнт має камеру. Браузер захоплює JPEG-кадр → POST → Pi face_recognition → токен підвищеної сесії. Фото не зберігається.
- **Голосовий відбиток** — ідентифікація під час голосових запитів (персоналізація команд, не вхід у UI)

> **HTTPS обов'язковий** для `getUserMedia()`. Без нього браузер не надає доступ до камери та мікрофона. Самопідписаний сертифікат генерується автоматично при ініціалізації.

### 9.3 Модель користувача (SQLite)

```sql
user_id        TEXT PRIMARY KEY   -- uuid4
username       TEXT UNIQUE        -- login name
display_name   TEXT               -- display name
role           TEXT               -- admin | resident
pin_hash       TEXT               -- SHA256 PIN (salted)
created_at     REAL               -- unix timestamp
last_seen      REAL               -- last activity timestamp
face_enrolled  INTEGER DEFAULT 0
voice_enrolled INTEGER DEFAULT 0
active         INTEGER DEFAULT 1  -- soft delete flag
```

Міграція: застарілу роль `owner` нормалізовано до `admin`, застарілу таблицю `role_config` видалено.

### 9.4 Журнал аудиту

- Зберігається локально в SQLite. Доступний тільки `admin`.
- Що логується: вхід/вихід, голосові команди (текст запиту), зміни налаштувань, встановлення/видалення модулів, керування пристроями.
- Ротація: останні 10 000 записів.
- Сторінка `/settings/audit` у ui-core: таблиця з фільтрами за користувачем, дією, датою.

## 10. Мережа, безпека, віддалений доступ

### 10.1 Tailscale — віддалений доступ з інтернету

Tailscale встановлюється як системний модуль `remote-access`. Він створює зашифрований WireGuard-тунель без відкритих портів на роутері.

- Налаштування: у майстрі (крок 8) або `/settings/remote` — QR-код → `tailscale.com` → авторизація
- Після підключення Pi доступний за адресою `100.x.x.x` або через MagicDNS (`smarthome-kitchen.ts.net`)
- Безкоштовний план Tailscale: до 100 пристроїв, без обмежень трафіку
- Статус: `/settings/remote` → "Підключено / Відключено / Помилка"

### 10.2 Брандмауер — правила iptables

```bash
# Core API — localhost only
iptables -A INPUT -p tcp --dport 7070 -s 127.0.0.1 -j ACCEPT
iptables -A INPUT -p tcp --dport 7070 -j DROP

# Web interface — local network + Tailscale
iptables -A INPUT -p tcp --dport 80 -s 192.168.0.0/16 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -s 100.0.0.0/8 -j ACCEPT  # Tailscale
iptables -A INPUT -p tcp --dport 80 -j DROP
```

Розділ `/secure` недоступний для користувацьких модулів — контейнери не мають монтування тому `/secure`.

### 10.3 Обмеження частоти запитів

| Дія | Ліміт | Наслідок |
|---|---|---|
| Неправильний PIN | 5 спроб / 60 сек | 10-хвилинне блокування, запис у журнал аудиту |
| Запити до Core API | 100 / сек на токен | HTTP 429 |
| WebSocket-аудіо (STT) | 1 сесія на користувача | Нове з'єднання відхиляється |
| З'єднання Module Bus | 1 на модуль | Дубльоване з'єднання відхиляється |

### 10.4 HTTPS та сертифікати

- Самопідписаний сертифікат (mkcert) генерується автоматично при ініціалізації
- Видається для `smarthome.local`, `smarthome-setup.local` та IP-адреси пристрою
- Користувач може завантажити власний сертифікат через `/settings/security`
- Без HTTPS — `getUserMedia()` недоступний. Це блокуюча вимога для голосу та Face ID.

## 11. Сповіщення

### 11.1 Канали доставки

| Канал | Коли працює | Реалізація |
|---|---|---|
| TTS-голос | Завжди (Pi вдома) | Piper → ALSA/BT. Пріоритет: критичні сповіщення. |
| SSE у браузері | Поки браузер відкритий | EventSource в ui-core. Статус у реальному часі. |

| Telegram-бот | Telegram встановлений | Модуль `INTEGRATION`. Авторизація через Bot API. |

### 11.2 Пріоритети сповіщень

| Рівень | Приклади | Канали |
|---|---|---|
| `CRITICAL` | Датчик пожежі, протікання, вторгнення | TTS негайно + Push + Telegram |
| `HIGH` | Батарея < 10%, Pi перегрівається | Push + Telegram |
| `NORMAL` | Завдання виконано, модуль оновлено | SSE у браузері |
| `INFO` | Світло вимкнено, двері зачинено | Тільки історія (без push) |

## 12. Імпорт з існуючих систем

### 12.1 Підтримувані системи (бета)

| Система | Авторизація | Що імпортується |
|---|---|---|
| **Home Assistant** | OAuth2 + URL сервера | Пристрої, кімнати, автоматизації (прості), сцени |
| **Tuya / SmartLife** | OAuth2 через посилання → QR у додатку | Пристрої, кімнати, DP-коди команд |
| **Philips Hue** | Натиснути кнопку на Bridge (локально) | Світильники, групи, сцени. Інтернет не потрібен. |

| IKEA TRÅDFRI | Автогенерація PSK | Світильники, групи, жалюзі |
| MQTT Broker | host + логін + пароль | Топіки як пристрої |

### 12.2 Процес імпорту (4 кроки в ui-core)

1. **Вибір джерела** — картки з логотипами, значок "Популярне" на перших трьох
2. **Авторизація** — OAuth: кнопка → редирект → callback. Кнопка Bridge: таймер 30 сек. PSK: форма.
3. **Попередній перегляд** — чекбокси за групами: Освітлення / Клімат / Безпека / Автоматизації
4. **Прогрес** — SSE-потік: "Імпорт 12/20...", "Конвертація автоматизацій 3/8 (5 потребують ручного налаштування)"

### 12.3 Модулі-мости

Після імпорту пристроями керують через модуль-міст — він транслює команди Core API назад у вихідну систему з двонаправленою синхронізацією стану.

- `ha-bridge` — WebSocket-синхронізація з Home Assistant у реальному часі
- `tuya-bridge` — Tuya Open API + push через Tuya MQTT

> **Автоматизації:** прості (`if X → Y`) конвертуються повністю. Складні (шаблони Jinja2, скрипти) → чернетки з позначкою "потребує ручного налаштування".

## 13. Моніторинг ресурсів та деградація

### 13.1 hw-monitor — системний модуль

- Кожні 30 сек: температура CPU (`/sys/class/thermal`), RAM (`free`), диск (`df`), час роботи
- Дані включаються в heartbeat-пінги до платформи SmartHome LK
- Графіки за останні 24 години на сторінці `/settings/system` у ui-core

### 13.2 Пороги та автоматичні дії

| Метрика | Поріг | Дія |
|---|---|---|
| Температура CPU | > 80°C | ⚠️ WARN-сповіщення користувачу + повідомлення платформі |
| Температура CPU | > 90°C | 🔴 Зупинка LLM Engine + CRITICAL-сповіщення |
| Вільна RAM | < 300 МБ | Блокування встановлення нових модулів |
| Вільна RAM | < 150 МБ | Зупинка AUTOMATION → зупинка INTEGRATION → попередження |
| Вільний диск | < 500 МБ | Попередження |
| Вільний диск | < 100 МБ | Зупинка бекапу |

### 13.3 Стратегія деградації при нестачі RAM

1. Попередження користувачу в ui-core + блокування встановлення нових модулів
2. При RAM < 150 МБ: автозупинка за пріоритетом — спочатку AUTOMATION, потім INTEGRATION
3. UI-модулі та DRIVER-модулі — тільки з явного дозволу користувача
4. SYSTEM-модулі ніколи не зупиняються (виняток: LLM Engine при перегріві CPU > 90°C)

### 14.2 Базовий клас модуля

```python
from sdk.base_module import SmartHomeModule, intent, on_event, scheduled

class MyModule(SmartHomeModule):
    name = "my-climate-module"
    version = "1.0.0"

    async def on_start(self):
        self.logger.info("Module started")

    @intent(r"set (?:the )?temperature to (\d+)")
    async def handle_temperature(self, match, context):
        temp = int(match.group(1))
        await self.devices.set_state(context.device_id, {"temperature": temp})
        return f"Temperature set to {temp}"

    @on_event("device.state_changed")
    async def handle_state(self, event):
        device = await self.devices.get(event.device_id)
        if device.state.get("temperature") > 25:
            await self.devices.set_state(device.id, {"fan": True})

    @scheduled("*/5 * * * *")
    async def periodic_check(self):
        devices = await self.devices.list(type="sensor")

    async def on_stop(self):
        pass  # graceful shutdown
```

```python
from my_module import MyModule

module = MyModule()
asyncio.run(module.start())
```

## 15. Офлайн-режим

> **Базовий сценарій "керування домом голосом та через UI" працює повністю без інтернету. Хмара — це опціональне розширення, а не обов'язкова залежність.**

| Функція | Без інтернету | Примітки |
|---|---|---|
| Голосовий асистент (STT/TTS) | ✅ Так | Vosk + Piper — повністю локально |
| LLM Intent Router | ✅ Так | Ollama локально на Pi 5 |
| Device Registry | ✅ Так | Локальна SQLite |
| Автоматизації | ✅ Так | Локальні пристрої |
| Веб-інтерфейс :80 | ✅ Так | Локальна мережа |
| Історія діалогів | ✅ Так | Локальна SQLite |

## 16. Критерії готовності v0.3

### 16.1 Онбординг

- [ ] Готовий образ .img можна записати на SD, і він завантажується без додаткового налаштування
- [ ] Pi створює точку доступу `SmartHome-Setup` при першому запуску. QR на HDMI, якщо підключений.
- [ ] Майстер налаштування завершує всі 9 кроків у браузері телефону без помилок
- [ ] Після майстра відображається екран "Що далі" з трьома рекомендаціями

### 16.2 Ядро та модулі

- [ ] Гібридна модульна система працює. Системні модулі завантажуються в процесі через importlib.
- [ ] Користувацькі модулі підключаються через WebSocket Module Bus та коректно обмінюються повідомленнями.
- [ ] Збій одного користувацького модуля не зупиняє інші (тест: `kill -9` на підпроцесі модуля)
- [ ] Watchdog: systemd + Docker автоматично перезапускають контейнери після збою
- [ ] Integrity Agent виявляє зміни файлів ядра протягом ≤ 30 сек
- [ ] Circuit breaker Module Bus спрацьовує після 5 збоїв за 60 сек

### 16.3 Голос та LLM

- [ ] STT працює без інтернету (тест: `ip link set eth0 down` → команда розпізнається)
- [ ] TTS вимовляє відповідь локально через Piper
- [ ] Режим приватності: GPIO-кнопка ТА голосова команда вимикають мікрофон
- [ ] Fast Matcher обробляє зареєстровані команди за < 1мс
- [ ] 4-рівневий Intent Router коректно диспетчеризує через усі рівні
- [ ] Біометричні дані відсутні в будь-яких вихідних HTTP-запитах (тест через `tcpdump`)

### 16.4 UI та доступ

- [ ] PWA встановлюється на головний екран телефону. Офлайн-сторінка показує кеш.
- [ ] Tailscale-тунель налаштовується через ui-core. Pi доступний через MagicDNS.
- [ ] Усі 4 режими UI (HEADLESS/KIOSK/FRAMEBUFFER/TTY) працюють коректно
- [ ] HTTPS: самопідписаний сертифікат, `getUserMedia()` доступний

### 16.5 Безпека

- [ ] Core API :7070 недоступний ззовні localhost (тест через зовнішній IP)
- [ ] 5 неправильних PIN → 10-хвилинне блокування, запис у журнал аудиту
- [ ] Журнал аудиту зберігає дії. Доступний тільки `admin`.
- [ ] Деградація RAM: AUTOMATION зупиняється при вільній RAM < 150 МБ

### 16.6 SDK та імпорт

- [ ] `smarthome new-module` створює робочу структуру
- [ ] `smarthome dev` запускає mock Core API та Module Bus локально
- [ ] Імпорт з Home Assistant: пристрої та прості автоматизації
- [ ] OAuth QR-процес завершується успішно для Tuya та Home Assistant
